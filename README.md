5-2. 사용 경험 기반 분석
✅ Make — "자유도 높은 세밀한 설계"
Make는 캔버스 위에 모듈을 배치하고 연결하는 방식으로, 워크플로우를 시각적으로 설계하는 느낌이 강하다. Router 하나로 여러 분기를 한 시나리오에서 처리할 수 있어 복잡한 로직 구현에 유리하다. Google Sheet의 새 행을 감지하는 방식도 세밀하게 설정할 수 있으며, 변수 매핑과 데이터 가공을 직접 다룰 수 있어 개발자 친화적인 도구에 가깝다. 단, 처음 접하는 사용자에게는 모듈 개념과 데이터 흐름 이해가 선행되어야 하므로 진입 장벽이 존재한다.

✅ Zapier — "직관적인 단계별 자동화"
Zapier는 Trigger → Filter → Action의 선형 구조로, 누구나 쉽게 자동화 흐름을 이해하고 구성할 수 있다. Google Sheet 트리거 설정도 드롭다운 방식으로 간단하게 완료되며, 각 단계가 명확히 구분되어 있어 실수가 적고 유지보수가 쉽다. 다만, 복잡한 분기 로직은 여러 Zap으로 분리해야 하므로 관리 포인트가 늘어나는 단점이 있다.

자동화 도구 핵심 개념 보고서
1. Trigger (트리거)
트리거는 자동화 흐름의 출발점이다. 특정 조건이나 이벤트가 발생했을 때 자동화를 시작하도록 신호를 보내는 역할을 한다. 즉, 트리거가 감지되기 전까지 자동화는 대기 상태에 머물며, 트리거가 발생하는 순간 비로소 전체 흐름이 작동하기 시작한다.

트리거의 유형은 크게 네 가지로 분류할 수 있다.

트리거 유형	설명	예시
이벤트 발생형	특정 이벤트가 일어날 때 작동	이메일이 수신되었을 때
시간 기반형	설정된 시간이 되면 작동	매일 오전 9시
데이터 변경형	데이터에 변화가 생길 때 작동	구글 시트에 새 행 추가 시
폼 제출형	사용자가 폼을 제출할 때 작동	설문지 또는 신청서 제출 시
트리거는 자동화의 시작을 책임지는 만큼, 어떤 트리거를 설정하느냐에 따라 자동화의 목적과 방향이 결정된다.

2. Action (액션)
액션은 트리거가 발생한 이후 실제로 수행되는 작업이다. 트리거가 "언제 시작할 것인가"를 정의한다면, 액션은 "무엇을 할 것인가"를 정의한다. 하나의 자동화 흐름 안에서 액션은 여러 개를 연속적으로 설정할 수 있으며, 각 액션은 순서대로 실행된다.

아래는 이메일 수신을 트리거로 설정했을 때의 액션 흐름 예시이다.


트리거: 새 이메일 수신
  └─ Action 1: 이메일 내용을 구글 시트에 자동 기록
  └─ Action 2: 슬랙 채널에 알림 메시지 전송
  └─ Action 3: 발신자에게 자동 회신 이메일 발송
이처럼 하나의 트리거에 여러 액션을 연결함으로써, 반복적으로 수행하던 업무를 자동화하여 시간과 인력을 절감할 수 있다.

3. 조건 분기 (Filter / Router)
조건 분기는 자동화 흐름 중간에 판단 기준을 삽입하여, 입력된 값이나 상황에 따라 흐름을 세분화하는 기능이다. 조건 분기를 활용하면 단순한 자동화를 넘어 상황에 맞는 유연한 대응이 가능해진다. 조건 분기는 크게 Filter와 Router 두 가지 방식으로 나뉜다.

Filter (필터)
필터는 설정한 조건을 만족하는 경우에만 다음 단계로 진행하고, 조건을 만족하지 않으면 자동화를 중단시키는 방식이다.


예시: 이메일 제목에 "긴급"이라는 단어가 포함된 경우에만 슬랙 알림 발송
      → 포함되지 않은 경우 자동화 흐름 중단
Router (라우터)
라우터는 조건에 따라 자동화 흐름을 여러 경로로 분기시키는 방식이다. 필터와 달리 흐름을 중단하지 않고, 각 조건에 맞는 경로로 연결하여 서로 다른 액션을 실행한다.


예시: 고객 등급에 따른 분기 처리
  ├─ VIP 고객    → 담당 매니저에게 직접 연결
  ├─ 일반 고객   → 챗봇 상담으로 연결
  └─ 신규 고객   → 서비스 안내 이메일 자동 발송
필터는 흐름의 통과 여부를 결정하고, 라우터는 흐름의 방향을 결정한다는 점에서 서로 다른 역할을 수행한다.

4. 자동화 도구 비교: Make vs Zapier
현재 가장 널리 사용되는 자동화 도구로는 Make(메이크) 와 Zapier(자피어) 가 있다. 두 도구 모두 다양한 앱과 서비스를 연결하여 자동화 흐름을 구성할 수 있지만, 설계 철학과 사용 방식에서 뚜렷한 차이를 보인다.

비교 항목	⚙️ Make	⚡ Zapier
사용 난이도	중~고급	초~중급
UI 방식	시각적 플로우차트 형태	단계별 리스트 형태
세밀한 설정	매우 강력하고 유연함	비교적 제한적
조건 분기	복잡한 다중 분기 가능	기본적인 수준
데이터 가공	함수 및 변환 기능 풍부	단순한 수준
연동 앱 수	1,000개 이상	6,000개 이상
가격	상대적으로 저렴	상대적으로 높음
적합한 사용자	복잡한 자동화 설계가 필요한 사용자	빠르고 간편한 자동화가 필요한 비개발자
Make를 선택해야 하는 경우
Make는 자동화 흐름을 시각적인 플로우차트 형태로 구성할 수 있어, 전체 흐름을 한눈에 파악하기 용이하다. 특히 데이터를 가공하거나 변환해야 하는 작업, 복잡한 조건 분기가 필요한 업무, 그리고 비용 효율을 중시하는 환경에서 강점을 발휘한다. 세밀한 설정이 가능한 만큼 초기 학습에 시간이 필요하지만, 익숙해지면 높은 자유도로 다양한 자동화를 구현할 수 있다.

Zapier를 선택해야 하는 경우
Zapier는 단계별 리스트 방식으로 자동화를 구성하기 때문에, 자동화 경험이 없는 사용자도 빠르게 적응할 수 있다. 연동 가능한 앱의 수가 6,000개 이상으로 Make보다 훨씬 많아, 다양한 서비스를 연결해야 하는 환경에서 유리하다. 복잡한 설정보다는 빠른 자동화 구현이 우선시되는 상황에 적합하다.

5. 자동화 흐름 단계별 설명
아래는 고객 문의 처리 업무를 자동화한 흐름의 예시이다. 구글 폼으로 접수된 문의를 자동으로 분류하고, 담당자에게 배정하여 고객에게 접수 확인 메시지를 발송하는 과정을 단계별로 나타낸다.


-자동화 흐름 예시-

[STEP 1] 🔔 TRIGGER — 자동화 시작
  └─ 고객이 구글 폼을 통해 문의를 제출함

[자피어] 🔍 FILTER — 조건 확인 및 필터링
  └─ 문의 유형이 "기술 지원"인가?
      ├─ YES → 다음 단계로 진행
      └─ NO  → 자동화 중단 (일반 문의는 별도 흐름에서 처리)

[메이크] 🔀 ROUTER — 조건 확인 및 필터링
  └─ 문의 유형이 "기술 지원"인가?
      ├─ YES → 다음 단계로 진행
      └─ NO  → 자동화 중단 (일반 문의는 별도 흐름에서 처리)


[STEP 4] ⚙️ ACTION 1 — 데이터 기록
  └─ 구글 시트에 문의 내용, 접수 시간, 담당자 자동 기록

[STEP 5] ⚙️ ACTION 2 — 고객 응대
  └─ 고객에게 "문의가 정상적으로 접수되었습니다" 자동 이메일 발송

[STEP 6] ⚙️ ACTION 3 — 현황 업데이트
  └─ 노션 데이터베이스에 처리 상태 "접수 완료"로 자동 업데이트
이처럼 자동화 흐름은 트리거를 시작으로, 필터와 라우터를 통해 상황에 맞게 분기되고, 각 경로에서 정의된 액션들이 순차적으로 실행되는 구조로 이루어진다. 이 흐름을 명확하게 설계할수록 오류 없이 안정적인 자동화를 구현할 수 있다.
